**Prior milestone:** v0.4 tabs (`e7e2d03`, package `0.10.0`)

---

## 1. Goal

Ship menu primitives: `<core-menu>` + `<core-menuitem>` + `<core-menu-separator>`. W3C ARIA menu pattern. Standalone-first design — consumers compose with `<core-popover>` (v0.8) for triggered menus. Keyboard nav (Arrow / Home / End / Enter / Space / Escape / single-character type-ahead). Submenu support via nested `<core-menu>` inside a menuitem.

Independent navigation primitive. No required dependency on v0.5 / v0.6 / v0.7 / v0.8 surfaces / v0.3 data-table / v0.4 tabs — the menu element itself is self-contained. Popover composition is a usage pattern, not an import.

---

## 2. Scope

### In scope (v0.9 "standard seed")

- `<core-menu>` — Shadow DOM container with state + ARIA + keyboard nav + type-ahead + submenu coordination
- `<core-menuitem>` — light DOM trigger with click delegation, slots for icon + label + trailing content
- `<core-menu-separator>` — light DOM divider with `role="separator"`
- Orientation: `vertical` (default) + `horizontal` (for menubar-style use)
- Keyboard navigation: Arrow keys (orientation-aware), Home, End, Enter, Space, Escape, type-ahead
- Disabled menuitems skipped in keyboard nav, click no-op
- Auto-wired ARIA: `role="menu|menuitem|separator"`, `aria-orientation`, `aria-disabled`, `aria-haspopup`, `aria-expanded`, roving `tabindex`
- Single-character type-ahead with 500ms accumulating window for multi-char refinement
- Submenu support: nested `<core-menu>` inside `<core-menuitem has-submenu>` opens via click / Enter / Space / ArrowRight (vertical); closes via ArrowLeft / Escape
- `core-menu-select` event with `{item, index, value}` detail
- `core-menu-close` cancellable event
- AX-rule example tests, integration tests, README + CLAUDE.md docs

### Out of scope (deferred to v0.9.1)

- `<core-menuitem role="menuitemcheckbox">` — toggle-able checkbox menu items
- `<core-menuitem role="menuitemradio">` — radio-group menu items (with `name` attribute)
- `<core-menubar>` — horizontal application menu bar at top of app with dropdown sub-menus
- `<core-menu-button>` convenience wrapper (button + popover + menu in one element)
- Submenu position-fix-on-overflow (auto-flip when submenu would exceed viewport)
- Type-ahead matching beyond `startsWith` (substring/fuzzy)
- Cancellable `core-menu-select` (preventDefault to roll back) — fire-and-forget in v0.9
- Animated open/close transitions for submenus

## 3. Architecture

### Layer position

v0.9 occupies its own "menu" tier. Sits on v0.2 foundation (`tokens`, `a11y`, `dom`). Independent of all prior tiers — the menu element has no required imports beyond Lit.

**Optional composition** with v0.8 popover for triggered menus is a documented usage pattern; the menu element itself does not import popover.

### Component model

```
LitElement
├── CoreMenu (Shadow DOM — chrome encapsulation)
│   State:
│     orientation: 'vertical'|'horizontal'    — attr (default 'vertical')
│     open: boolean                            — attr (for submenu visibility tracking)
│   Internal:
│     _items: CoreMenuitem[]                  (filtered from slot children, excluding separators)
│     _focusedIndex: number                   (-1 if no item focused)
│     _typeAheadBuffer: string
│     _typeAheadTimer: number | null
│     _activeSubmenu: CoreMenu | null         (currently-open child submenu)
│   Behaviour:
│     slotchange → re-read children, apply ARIA + roving tabindex, manage submenu hidden state
│     ArrowDown/Up (vertical) or ArrowLeft/Right (horizontal) → roving focus, skipping disabled
│     Home/End → first/last non-disabled item
│     Enter/Space → activate focused item (fire core-menu-select OR open submenu if has-submenu)
│     Escape → fire core-menu-close (cancellable)
│     ArrowRight (vertical, on submenu trigger) → open submenu + focus first item
│     ArrowLeft (vertical submenu) → close submenu + return focus to trigger
│     Single-char keydown → type-ahead match with 500ms accumulator
│
├── CoreMenuitem (Light DOM — appears in consumer's CSS)
│     Attrs (reflected): disabled, has-submenu, value
│     Properties (read-only, set by parent):
│       selected: boolean (from `data-focused` attribute)
│       index: number (from `data-menuitem-index` attribute)
│     ARIA (set by parent):
│       role="menuitem", aria-disabled, aria-haspopup, aria-expanded, tabindex
│     Click handler dispatches core-menuitem-click (internal) → parent owns selection
│     Slots: default (label), start (leading icon), end (trailing content)
│
└── CoreMenuSeparator (Light DOM)
      ARIA (set by parent): role="separator", aria-orientation
      Not focusable, not activatable
```

### Why this split

Matches the W3C ARIA menu pattern. `<core-menu>` is the container (Shadow DOM here to encapsulate the menu chrome — background, border, shadow, separator styling). `<core-menuitem>` + `<core-menu-separator>` are light DOM so consumers can apply their own CSS to menu item content without Shadow DOM piercing.

### File structure

```
src/menu/
├── menu.ts                # <core-menu> Shadow DOM container
├── menu.test.ts
├── menuitem.ts            # <core-menuitem> light DOM trigger
├── menuitem.test.ts
├── menu-separator.ts      # <core-menu-separator> light DOM divider
├── menu-separator.test.ts
├── index.ts               # re-exports + side-effect imports (children before parent)
└── menu_example_test.ts   # AX rule (4 examples)
```

## 4. Component APIs

### 4.1 `<core-menu>`

```html
<core-menu orientation="vertical">
  <core-menuitem>New file</core-menuitem>
  <core-menuitem>Open…</core-menuitem>
  <core-menuitem disabled>Open recent</core-menuitem>
  <core-menu-separator></core-menu-separator>
  <core-menuitem value="save">
    <core-icon slot="start" name="save"></core-icon>
    Save
    <span slot="end" class="shortcut">⌘S</span>
  </core-menuitem>
  <core-menuitem has-submenu>
    Export
    <core-menu>
      <core-menuitem>As PDF</core-menuitem>
      <core-menuitem>As HTML</core-menuitem>
    </core-menu>
  </core-menuitem>
</core-menu>
```

**Attributes (reflected):**
- `orientation="vertical|horizontal"` (default `vertical`)
- `open` (boolean) — for submenu visibility tracking; consumer can use for standalone menu show/hide as well

**Properties (read-only getters):**
- `items: CoreMenuitem[]` — current menuitem children, excluding separators
- `focusedIndex: number` — index of currently-focused item, `-1` if none

**Methods:**
- `focusFirst(): void` — focus first non-disabled item
- `focusLast(): void` — focus last non-disabled item
- `focusItem(indexOrEl: number | CoreMenuitem): void` — focus by index or element ref
- `close(): void` — closes any open child submenu; if this menu is a submenu, fires `core-menu-close` to parent

**Events:**
- `core-menu-select` — fired when a menuitem is activated. Detail: `{ item: CoreMenuitem, index: number, value: string }` where `value = item.getAttribute('value') || item.textContent.trim()`. Not cancellable in v0.9.
- `core-menu-close` — cancellable; fired when ESC pressed or `close()` called. preventDefault keeps menu open.

**Slots:** default — accepts `<core-menuitem>` and `<core-menu-separator>` children in any order.

**Parts:**
- `menu` — outer container with background, border, shadow

### 4.2 `<core-menuitem>`

```html
<core-menuitem disabled>Open recent</core-menuitem>

<core-menuitem value="save">
  <core-icon slot="start" name="save"></core-icon>
  Save
  <span slot="end" class="shortcut">⌘S</span>
</core-menuitem>

<core-menuitem has-submenu>
  Export
  <core-menu>
    <core-menuitem>As PDF</core-menuitem>
  </core-menu>
</core-menuitem>
```

Light DOM. Click delegates to parent `<core-menu>`.

**Attributes (reflected):**
- `disabled` (boolean) — non-interactive, skipped in keyboard nav
- `has-submenu` (boolean) — visual + ARIA cue; parent treats this item as a submenu trigger
- `value` (optional string) — payload for `core-menu-select`; defaults to `textContent.trim()` if absent

**Properties (read-only, set by parent):**
- `selected: boolean` — true when focused (mirrors `data-focused` attribute)
- `index: number` — slot position (raw, including separators) from `data-menuitem-index` attribute

**ARIA wiring (set by parent):**
- `role="menuitem"`
- `aria-disabled="true"` when `disabled`
- `aria-haspopup="menu"` when `has-submenu`
- `aria-expanded="true|false"` when has submenu and parent tracks state
- `tabindex="0"` if focused, `tabindex="-1"` otherwise (roving)

**Slots:**
- default — label text
- `start` — leading icon
- `end` — trailing content (chevron, shortcut hint, badge)

**Internal event:** `core-menuitem-click` (bubbles, composed) on click when not disabled. Parent `<core-menu>` listens and owns the user-facing `core-menu-select` dispatch.

### 4.3 `<core-menu-separator>`

```html
<core-menu-separator></core-menu-separator>
```

Light DOM. Pure visual divider.

**ARIA wiring (set by parent):**
- `role="separator"`
- `aria-orientation` matches parent menu's orientation

**Not focusable, not activatable.** Keyboard nav skips entirely.

**Visual:** parent's Shadow DOM CSS targets `core-menu-separator` (or `[role="separator"]`) with `border-top: 1px solid var(--core-menu-border-color); margin: 4px 0;` (vertical menu) or rotated for horizontal.

## 5. Behaviour details

### 5.1 Slotchange handling

`<core-menu>` observes the default slot. On every change:

1. Re-read all `<core-menuitem>` + `<core-menu-separator>` children
2. For each menuitem: apply `role="menuitem"`, `aria-disabled` (if disabled), `aria-haspopup="menu"` (if `has-submenu`), `aria-expanded="false"` (if has submenu)
3. For each separator: apply `role="separator"` + `aria-orientation` matching parent
4. Apply roving tabindex — first non-disabled menuitem gets `tabindex="0"`, others `tabindex="-1"`
5. For items with `has-submenu`: detect nested `<core-menu>` (direct child) via `Array.from(this.children).filter(...)` pattern (happy-dom-safe per v0.4 deviation), set `hidden` attribute on the submenu initially
6. Set `data-menuitem-index="{i}"` on each menuitem (excluding separators in the count) for the `index` getter

### 5.2 Type-ahead

W3C menu pattern: single-character type-ahead with multi-char refinement window.

**State per menu instance:**
- `_typeAheadBuffer: string` — accumulating chars
- `_typeAheadTimer: number | null` — setTimeout ID

**On keydown with single-character `key` (and no Ctrl/Meta/Alt):**
1. Append char to `_typeAheadBuffer`
2. Clear existing timer if set; set new 500ms timer to reset `_typeAheadBuffer = ''`
3. Find next non-disabled menuitem whose `textContent.trim().toLowerCase()` starts with `_typeAheadBuffer.toLowerCase()`, starting search from `_focusedIndex + 1`, wrapping to start if not found
4. If found: focus that item

**Examples:**
- Type `o`: focuses next item starting with "o" (Open)
- Type `o`, then `p` within 500ms: focuses next item starting with "op" (Open File over Open Recent if both exist)
- Type `o`, wait 500ms, type `s`: timer reset; now matches just "s" (Save)

### 5.3 Submenu open/close flow

**Open** (focused menuitem has `has-submenu`, user triggers):
- Triggers: click, Enter, Space, ArrowRight (vertical menu) / ArrowDown (horizontal menu)
- Behaviour:
  1. Find the nested `<core-menu>` (direct child of the menuitem)
  2. Set `aria-expanded="true"` on the trigger menuitem
  3. Remove `hidden` from the nested menu
  4. Store nested menu as `_activeSubmenu`
  5. Call `nestedMenu.focusFirst()` to move focus into submenu
  6. Nested menu's keyboard handlers take over (event bubbling stops at nested menu's host)

**Close** (in a submenu):
- Triggers: Escape, ArrowLeft (vertical) / ArrowUp (horizontal), focus leaves menu tree
- Behaviour:
  1. Set `aria-expanded="false"` on the parent trigger menuitem
  2. Set `hidden` on this nested menu
  3. Parent clears `_activeSubmenu = null`
  4. Focus returns to the parent trigger menuitem

**Single submenu at a time per level:** opening a different sibling's submenu closes the current `_activeSubmenu` first.

**Close cascade on Escape:**
- ESC at deepest level closes that submenu only
- Parent menu still open, focus returns to parent trigger
- Subsequent ESC closes next level up
- ESC at root menu fires `core-menu-close`

### 5.4 Keyboard nav (full table)

| Key | Vertical menu | Horizontal menu | Effect |
|---|---|---|---|
| `ArrowDown` | next item | open submenu (if focused has-submenu) | focus next non-disabled item |
| `ArrowUp` | prev item | — | focus prev non-disabled item |
| `ArrowRight` | open submenu (if focused has-submenu) | next item | open submenu or focus next |
| `ArrowLeft` | close submenu (if in submenu) | prev item | close submenu or focus prev |
| `Home` | first item | first item | focus first non-disabled item |
| `End` | last item | last item | focus last non-disabled item |
| `Enter` | activate focused | activate focused | fire `core-menu-select` or open submenu |
| `Space` | activate focused | activate focused | same as Enter |
| `Escape` | fire `core-menu-close` | fire `core-menu-close` | cancellable; if not cancelled, close menu |
| `<character>` | type-ahead | type-ahead | focus next item matching accumulated chars |

Wrap at boundaries skipping disabled items. If all items disabled, keyboard nav is a no-op.

### 5.5 Standalone vs popover-composed usage

**Standalone:**
```html
<core-menu>
  <core-menuitem>Dashboard</core-menuitem>
  <core-menuitem>Settings</core-menuitem>
</core-menu>
```
Always-visible panel. Consumer styles position + sizing.

**Popover-composed (triggered menus):**
```html
<core-button id="more-btn">More</core-button>
<core-popover anchor="#more-btn" placement="bottom-start">
  <core-menu>
    <core-menuitem>Edit</core-menuitem>
    <core-menuitem>Delete</core-menuitem>
  </core-menu>
</core-popover>
```
v0.8 popover handles anchor positioning + light-dismiss + focus. Consumer listens for both `core-menu-select` (action) and `core-popover-close` (dismissal).

Recommended UX pattern: consumer calls `popover.hide()` from the `core-menu-select` handler (close menu after action).

### 5.6 Focus management

Roving tabindex pattern (same as v0.4 tabs):
- Only one menuitem has `tabindex="0"` at any time — the focused one
- All others `tabindex="-1"`
- Keyboard nav moves focus within the menu without leaving the tab order
- Initial state: first non-disabled menuitem gets `tabindex="0"`

When menu is opened programmatically (e.g. via popover trigger), consumer should call `menu.focusFirst()` to give the menu keyboard focus.

When all items are disabled (degenerate case), no item gets `tabindex="0"` — keyboard nav is a no-op.

### 5.7 Accessibility

All ARIA attributes set automatically by the parent `<core-menu>` based on state. Consumer doesn't need to wire anything.

| Element | Role | Required ARIA |
|---|---|---|
| `<core-menu>` shadow `[part="menu"]` | `menu` | `aria-orientation="vertical\|horizontal"` |
| `<core-menuitem>` | `menuitem` | `aria-disabled`, `aria-haspopup`, `aria-expanded`, `tabindex` (roving) |
| `<core-menu-separator>` | `separator` | `aria-orientation` matches parent |

`aria-orientation` on the menu reflects the `orientation` attribute. Screen readers announce keyboard nav direction correctly.

### 5.8 happy-dom guardrails

- `slotchange` fires correctly ✓ — primary mechanism for child observation
- `MutationObserver` ✓ — backup for child attribute mutations
- `:scope >` CSS selector unreliable — use `Array.from(this.children).filter(...)` pattern (v0.4 T4 deviation)
- Lit `@property({ reflect: true })` is async — supplement with explicit `setAttribute` when state must be synchronously observable (v0.3 T6 / v0.4 T2 patterns)
- Keyboard event dispatch ✓ — type-ahead simulation works with synthetic keydown events
- Native `<input>` not used (menu items use native button-like behaviour via `click()`)
- Native `focus()` calls work but `activeElement` tracking limited — assert via `tabindex` attribute presence, not `document.activeElement` identity
- Submenu visibility via native `hidden` attribute — happy-dom honours it (display: none semantics)

Every behaviour relying on real-browser semantics (actual focus moves, real keyboard tab cycle entering submenus) is gated with `if (...) return;` early-returns rather than `it.skip()`. Each early-return commented `// Playwright sweep covers …`.
