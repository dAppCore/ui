# CoreUI v0.3 — Data Table (`<core-data-table>` + `<core-column>`) Design

**Status:** Approved (brainstorm 2026-05-17)

**Plan file:** `docs/superpowers/plans/2026-05-17-coreui-v0.3-data-table.md` (to be written)

**Prior milestone:** v0.8 surfaces (`b0fa45c`, package `0.8.0`)

---

## 1. Goal

Ship the data-presentation tier of CoreUI — `<core-data-table>` + `<core-column>` — the RFC §8 seed component that's been deferred since the start. Useful day-one: render rows, sortable columns, pagination, row selection, density + sticky-header chrome, loading + empty states. Brand-neutral, Shadow DOM, zero deps beyond Lit.

Unblocks data-heavy app UIs without forcing consumers to bolt on AG Grid, TanStack Table, or similar.

---

## 2. Scope

### In scope (v0.3 "full seed")

- `<core-data-table>` — Shadow DOM host with all state
- `<core-column>` — light-DOM metadata-only child element
- Two pure utilities: `sort.ts`, `pagination.ts`
- Client-side sort (built-in, type-aware) with consumer override via cancellable event
- Built-in pagination footer with slot override
- Single + multi row selection with tri-state header checkbox
- Density modes (`comfortable | cozy | compact`)
- Sticky header (on by default)
- Loading + empty states (slot + default text)
- Type-aware default rendering (`text | number | date | boolean`)
- Custom per-column render via `column.render` property
- Keyboard navigation (arrow + space + enter)
- ARIA roles (`table`, `rowgroup`, `row`, `columnheader`, `cell`, `aria-sort`)
- AX-rule example tests, integration tests, README + CLAUDE.md docs

### Out of scope (deferred)

- **Virtualization** — IntersectionObserver-based row windowing. Defer to v0.3.1; complex enough to be its own slice
- **Column resize** — drag handle on column borders. Defer
- **Column reorder** — drag-drop column headers. Defer
- **Filter UI** — built-in per-column filter inputs. Defer (consumer can build via header slot for v0.3)
- **Inline editing** — click-to-edit cell. Defer
- **Tree/nested rows** — expandable parent rows. Defer
- **Server-side rendering optimisations** — Lit SSR helpers. Defer
- **Bulk action toolbar** — appears above table when rows selected. Defer; consumer can build via header slot
- **Sticky columns** (frozen first/last columns) — defer
- **Row drag-drop reorder** — defer
- **Selection event cancellation** — `core-selection-change` is fire-and-forget in v0.3; cancellable variant deferred to v0.3.1

## 3. Architecture

### Layer position

v0.3 occupies its own "data presentation" tier. Sits on the v0.2 foundation (`tokens`, `colour`, `a11y`, `dom`, `math`). Independent of v0.5 primitives, v0.6 router, v0.7 forms, v0.8 surfaces.

**Optional consumer-side composition** — examples may show using `<core-button>` for pagination controls or `<core-checkbox>` for selection cells, but the table doesn't import those components. It uses native HTML elements internally so consumers without other CoreUI tiers can still use the table.

### Component model

```
LitElement
├── CoreDataTable (Shadow DOM)
│   State:
│     rows: Row[]                         — data array (property)
│     selected: Set<string>               — selected row keys (property)
│     page: number                        — current page index (property)
│     sortBy: string | null               — current sort column key (attr)
│     sortDir: 'asc' | 'desc'             — current sort direction (attr)
│     loading: boolean                    — attr
│     keyField: string                    — attr (row property for stable keys; default 'index')
│   Behaviour:
│     Reads <core-column> children via MutationObserver
│     Renders shadowed table chrome
│     Built-in sort (cancellable event for server-driven)
│     Built-in pagination footer (slot override available)
│     Selection state machine (none / single / multi)
│     Keyboard nav (arrow / space / enter)
│
└── CoreColumn (light DOM, metadata-only — never renders)
    Attributes:
      key, label, sortable, type, align, width, hidden
    Property:
      render: (row, ctx) => TemplateResult | string | undefined
```

Pattern is similar to `<core-radio-group>` + `<core-radio>` from v0.7: the parent owns all behaviour; children are declarative metadata read by the parent.

### Why Shadow DOM

Encapsulates table chrome (sticky header CSS, density mode selectors, hover/selected row backgrounds) and exposes `::part()` API for consumer theming. Same rationale as v0.7 forms and v0.8 surfaces.

## 4. Component APIs

### 4.1 `<core-data-table>`

```html
<core-data-table
  .rows=${this.users}
  selection="multi"
  page-size="10"
  density="cozy"
  sticky-header
  key-field="id"
>
  <core-column key="name"    label="Name"     sortable></core-column>
  <core-column key="email"   label="Email"></core-column>
  <core-column key="created" label="Created"  sortable type="date"></core-column>
  <core-column key="score"   label="Score"    sortable type="number" align="end"></core-column>
  <core-column key="actions" label="" width="80px"></core-column>

  <div slot="empty">No users match this filter.</div>
  <div slot="loading">Loading users…</div>
</core-data-table>

<script>
  const actionsCol = document.querySelector('core-column[key="actions"]');
  actionsCol.render = (row) => html`
    <button onclick="editUser('${row.id}')">Edit</button>
  `;
</script>
```

**Attributes (reflected unless noted):**

- `selection="none|single|multi"` (default `none`)
- `page-size` (number; if > 0, pagination footer renders; if unset/0, all rows shown)
- `density="comfortable|cozy|compact"` (default `cozy`)
- `sticky-header` (boolean, default `true`; opt-out via `sticky-header="false"`)
- `loading` (boolean; shows loading slot + dims body)
- `sort-by` (string — column key; bidirectional, reflects on sort changes)
- `sort-dir="asc|desc"` (default `asc` when sort-by set)
- `key-field` (string; row property name for stable keys; default `'index'`)
- `preserve-selection` (boolean; if set, selected keys persist across `rows` setter changes)
- `preserve-page` (boolean; if set, page index persists across `rows` setter changes)

**Properties (not reflected — too dynamic):**

- `rows: Row[]` — data array; setter triggers re-render + selection/page pruning unless `preserve-*` set
- `selected: Set<string>` — selected row keys; consumer reads/writes
- `page: number` — current page index (0-based); setter triggers re-render
- `totalRows: number` (read-only getter) — `rows.length`, exposed for custom pagination slot consumers

**Methods:**

- `refresh()` — force re-read of `<core-column>` children + re-render (useful after dynamically adding columns)
- `clearSelection()` — empties `selected`, fires `core-selection-change`
- `selectAll()` — multi-select only; adds all visible-page row keys to `selected`
- `selectNone()` — alias for `clearSelection()`
- `sortBy(key: string, dir?: 'asc' | 'desc')` — programmatic sort; if `dir` omitted, toggles

**Events:**

- `core-sort-change` — cancellable; detail: `{ key: string | null, dir: 'asc' | 'desc' | null, previousKey: string | null, previousDir: 'asc' | 'desc' | null }`
- `core-page-change` — detail: `{ page: number, pageSize: number, totalRows: number }`
- `core-selection-change` — detail: `{ selected: Set<string>, added: string[], removed: string[] }`
- `core-row-click` — detail: `{ row: Row, key: string, event: MouseEvent | KeyboardEvent }`

**Slots:**

- default — `<core-column>` children (and optionally other markup that's ignored)
- `header` — content above the table chrome (filter bar, title, bulk-actions toolbar)
- `footer` — content below the table body, above default pagination
- `pagination` — full override of built-in pagination widget (consumer reads `el.page`/`el.pageSize`/`el.totalRows` and fires their own `core-page-change`)
- `loading` — shown when `loading` attribute set; default fallback: "Loading…"
- `empty` — shown when `rows.length === 0 && !loading`; default fallback: "No data."

**Parts:**

- `table` — outer table element (or div with `role="table"`)
- `header` — table header section
- `header-row` — header row
- `header-cell` — individual header cell (one per column)
- `body` — table body section
- `row` — every data row
- `row-selected` — selected rows (additional state-part)
- `cell` — individual data cell
- `pagination` — pagination footer wrapper
- `loading` — loading state overlay
- `empty` — empty state container

### 4.2 `<core-column>`

```html
<core-column key="name" label="Name" sortable type="text" align="start" width="200px"></core-column>
```

Light-DOM, metadata-only. Never renders anything itself — `<core-data-table>` reads its attributes and `render` property.

**Attributes (reflected):**

- `key` — row property name to read (required unless `render` set)
- `label` — header text (default: `key` value)
- `sortable` (boolean) — clickable header with sort indicator + ARIA
- `type="text|number|date|boolean"` (default `text`) — drives default formatting + sort comparator
- `align="start|center|end"` (default `start`)
- `width` — CSS width value (e.g. `"200px"`, `"20%"`, `"1fr"`); unset = `auto`
- `hidden` (boolean) — column not rendered

**Properties (JS-only):**

- `render: (row: Row, ctx: { rowIndex, columnIndex, isSelected }) => TemplateResult | string | undefined`
  - Default if not set: type-aware formatting of `row[key]` (see §4.3)
  - Returning `undefined` falls back to default render
- `sortFn: (a: Row, b: Row, dir: 'asc' | 'desc') => number` — custom comparator (overrides type-default)

### 4.3 Type-aware defaults

| `type` | Default render | Default sort comparator |
|---|---|---|
| `text` | `String(row[key])` | locale-aware via `String(a[key]).localeCompare(String(b[key]))` |
| `number` | `Number(row[key]).toLocaleString()` | numeric `Number(a[key]) - Number(b[key])` |
| `date` | `new Date(row[key]).toLocaleDateString()` | epoch `+new Date(a[key]) - +new Date(b[key])` |
| `boolean` | `row[key] ? '✓' : '—'` | `Number(Boolean(a[key])) - Number(Boolean(b[key]))` |

All consumer-overridable: `column.render = …` overrides display; `column.sortFn = …` overrides comparator.

## 5. Behaviour details

### 5.1 Sort

**Click flow:**
1. User clicks sortable column header
2. Compute next state — tri-state cycle:
   - Currently unsorted on this column → sort `asc`
   - Currently `asc` on this column → sort `desc`
   - Currently `desc` on this column → unsorted (restore original order)
   - Different column was sorted → sort `asc` on new column
3. Fire `core-sort-change` (cancellable) with `{ key, dir, previousKey, previousDir }`
4. If not `preventDefault()`-ed: apply built-in sort to internal `rows` snapshot and re-render
5. Update `sort-by` + `sort-dir` reflected attributes (regardless of cancel; consumer typically uses these for UI state)

**Server-driven path:** consumer listens for `core-sort-change`, calls `ev.preventDefault()`, fetches new sorted data from server, and assigns `el.rows = newData`. Sort indicator already updated via reflected attrs.

**Original-order preservation:** internal `_originalRows` snapshot taken when `rows` setter assigned. The tri-state "unsorted" returns to this order. New `rows` assignment refreshes the snapshot.

**Sort utility (`_shared/sort.ts`) API:**

```ts
export type SortDir = 'asc' | 'desc';
export type ColumnType = 'text' | 'number' | 'date' | 'boolean';

export function getComparator(type: ColumnType, dir: SortDir):
  (a: unknown, b: unknown) => number;

export function sortRows<T>(
  rows: T[],
  key: string,
  type: ColumnType,
  dir: SortDir,
  sortFn?: (a: T, b: T, dir: SortDir) => number,
): T[];  // returns new array, non-mutating
```

### 5.2 Pagination

**When active:** `page-size` attribute is set and > 0.

**Visible rows computation:** apply sort first, then `pageSlice(sortedRows, page, pageSize)`.

**Default pagination footer:**
```
                                              Showing 11–20 of 47   [‹] [1] [2] [3] [›]
```

- Prev/next disabled at boundaries
- Page numbers windowed: current ±2 + first/last with `…` gaps via `pageWindow()`
- Range display: `"Showing {start}–{end} of {total}"`
- Page button active state: visual highlight + `aria-current="page"`

**Events:** `core-page-change` fires on prev/next/page-button click. Detail: `{ page, pageSize, totalRows }`.

**Slot override:** `<div slot="pagination">…</div>` replaces default widget entirely. Consumer reads `el.page` / `el.pageSize` / `el.totalRows` and dispatches their own `core-page-change` event from inside the slot, OR sets `el.page` directly. Slotted pagination always reflects current state.

**Reset on `rows` change:** page resets to 0 unless `preserve-page` attr set. Useful for "user typed in filter → re-paginate from start".

**Pagination utility (`_shared/pagination.ts`) API:**

```ts
export function pageCount(totalRows: number, pageSize: number): number;
export function pageSlice<T>(rows: T[], page: number, pageSize: number): T[];
export function pageWindow(
  currentPage: number,
  totalPages: number,
  windowSize: number,  // pages each side of current; default 2
): (number | 'gap')[];
```

`pageWindow(2, 10, 2)` → `[0, 1, 2, 3, 4, 'gap', 9]`.

### 5.3 Selection

**`selection="single"`:**
- First column rendered as a radio input (native `<input type="radio" name="core-table-{uid}">`)
- Only one row's key in `selected: Set<string>` at a time
- Header cell of selection column is empty (no select-all)
- Clicking anywhere on a row also selects (consumer can opt out via `no-row-click-select` attr)

**`selection="multi"`:**
- First column rendered as a checkbox (native `<input type="checkbox">`)
- Header has select-all checkbox with tri-state via `indeterminate` DOM property:
  - none selected → unchecked
  - all visible-page selected → checked
  - some selected → indeterminate
- Click on row does NOT auto-select (avoids accidents); only the checkbox toggles

**Programmatic API:**
- `el.selectAll()` — adds all visible-page row keys
- `el.selectNone()` / `el.clearSelection()` — empties `selected`
- `el.selected = new Set(['id-1', 'id-2'])` — direct assignment

**Persistence:**
- On `rows` change: prune keys no longer present (unless `preserve-selection` set)
- `preserve-selection` use case: server-paginated table where rows on page 1 are gone after navigating to page 2, but user's selection should persist across pages

**Event:** `core-selection-change` fires on every mutation. Detail: `{ selected: Set<string>, added: string[], removed: string[] }`. Fire-and-forget in v0.3 (no cancellation).

### 5.4 Chrome

**Sticky header (default on):**
```css
:host([sticky-header]) [part="header"] {
  position: sticky;
  top: 0;
  z-index: 1;
  background: var(--core-table-header-bg);
}
```

Requires the Shadow body to be scrollable (consumer sets `max-height` on the host or wraps in a container; documented in usage).

**Density:**
```css
:host                          { --pad: var(--core-table-cell-padding-cozy); }
:host([density="comfortable"]) { --pad: var(--core-table-cell-padding-comfortable); }
:host([density="compact"])     { --pad: var(--core-table-cell-padding-compact); }

[part="cell"], [part="header-cell"] { padding: var(--pad); }
```

**Hover + selected row backgrounds:**
```css
[part="row"]:hover               { background: var(--core-table-row-bg-hover); }
[part="row"][data-selected]      { background: var(--core-table-row-bg-selected); }
[part="row"][data-selected]:hover {
  background: color-mix(in oklch,
    var(--core-table-row-bg-selected),
    var(--core-table-row-bg-hover) 50%);
}
```

### 5.5 Loading + empty states

- `loading` attr present → body dimmed (`opacity: 0.5; pointer-events: none`); `<slot name="loading">` rendered overlaid centred (fallback: `<div>Loading…</div>`)
- `rows.length === 0 && !loading` → `<slot name="empty">` rendered in body area (fallback: `<div>No data.</div>`)
- States are mutually exclusive — `loading` wins

### 5.6 Keyboard navigation

Each row gets `tabindex="0"` (or `-1` for non-focused rows; rover-tab pattern):
- Arrow Up / Down — move focus to prev/next visible row (wraps at boundaries)
- Space — toggle selection (when `selection !== "none"`)
- Enter — fire `core-row-click` with the focused row
- Home / End — first / last visible row

Sortable header cells are tabbable (`tabindex="0"`):
- Enter / Space — trigger sort (same as click)

### 5.7 Accessibility

- `role="table"` on `[part="table"]`
- `role="rowgroup"` on `[part="header"]` and `[part="body"]`
- `role="row"` on each row
- `role="columnheader"` on header cells; `aria-sort="ascending|descending|none"` on sortable columns reflecting current state
- `role="cell"` on data cells
- `aria-rowcount` on `[part="table"]` reflecting `totalRows` (so screen readers announce total even when paginated)
- `aria-rowindex` on each row (1-based, accounting for pagination — first visible row on page 2 of 10 = rowindex 11)
- `aria-selected="true|false"` on rows when `selection !== "none"`
- Selection inputs have `aria-label` ("Select row {keyValue}" / "Select all rows on this page")

### 6.3 happy-dom guardrails

- `MutationObserver` exists, fires on attribute changes — column attribute observation works
