## 1. Goal

Ship tabbed-interface primitives: `<core-tabs>` + `<core-tab>` + `<core-tabpanel>`. Standard W3C ARIA APG tablist pattern. Horizontal + vertical orientation. Keyboard nav (arrow / Home / End). Automatic + manual activation modes. Disabled tabs skipped. Active indicator with smooth slide. Auto-wired ARIA. Zero deps beyond Lit.

Independent navigation primitive — no dependency on v0.5 / v0.6 / v0.7 / v0.8 / v0.3 tiers. Day-one useful for any app with multi-section settings, dashboards, or content panels.

---

## 2. Scope

### In scope (v0.4 "standard seed")

- `<core-tabs>` — Shadow DOM container owning state + tablist chrome + indicator
- `<core-tab>` — light DOM trigger with click + ARIA wiring
- `<core-tabpanel>` — light DOM panel with `hidden` toggle + ARIA wiring
- Pairing API: implicit by index OR explicit `for`/`id` (both, dual-resolution)
- Horizontal + vertical orientation
- Keyboard navigation: ArrowLeft/Right (horizontal) or ArrowUp/Down (vertical), Home, End
- Automatic activation (W3C default) + manual activation (`activation="manual"` attr)
- Disabled tabs (`<core-tab disabled>`) skipped in keyboard nav, click no-op
- Active indicator (sliding underline/edge) via CSS transition
- Auto-wired ARIA: `role="tablist|tab|tabpanel"`, `aria-selected`, `aria-controls`, `aria-labelledby`, `aria-orientation`, roving `tabindex`
- Auto-generated stable IDs for unpaired panels
- `core-tab-change` cancellable event
- AX-rule example tests, integration tests, README + CLAUDE.md docs

### Out of scope (deferred to v0.4.1)

- **Lazy panel rendering** — inactive panels rendered + hidden by default. Lazy option deferred.
- **Scrollable overflow** — when tabs exceed viewport width, scroll horizontally with prev/next arrows. Deferred.
- **Drag-reorder** — drag tabs to reorder. Deferred.
- **Closeable tabs** — `<core-tab closeable>` with × button. Deferred.
- **Add-tab UI** — `+` button to dynamically append tabs. Deferred (consumer can append `<core-tab>` + `<core-tabpanel>` manually + call `refresh()`).
- **Indicator variants** — only sliding underline in v0.4. Pill, segmented, none — deferred (consumer can override via `::part(indicator) { display: none }` for now).

## 3. Architecture

### Layer position

v0.4 occupies its own "navigation primitives" tier. Sits on the v0.2 foundation (`tokens`, `colour`, `a11y`, `dom`). Independent of v0.5 primitives, v0.6 router, v0.7 forms, v0.8 surfaces, v0.3 data-table.

### Component model

```
LitElement
├── CoreTabs (Shadow DOM — chrome encapsulation for tablist + indicator)
│   State:
│     selectedIndex: number          — attr 'selected-index' (default 0)
│     activation: 'auto'|'manual'    — attr (default 'auto')
│     orientation: 'horizontal'|'vertical'  — attr (default 'horizontal')
│   Internal:
│     _tabs: CoreTab[]              (read from slot on slotchange)
│     _panels: CoreTabpanel[]
│     _pairs: Array<{tab, panel}>   (resolved per §3.1)
│     _uid: number                  (per-instance counter for auto-id)
│   Behaviour:
│     slotchange → re-read children + re-pair + apply ARIA + reposition indicator
│     ArrowKey / Home / End → roving focus + (auto-mode) activate
│     Space / Enter on focused tab → activate (manual mode only)
│
├── CoreTab (Light DOM — appears in consumer's CSS)
│     Attrs (reflected): for, disabled
│     Properties (read-only, set by parent):
│       selected: boolean
│       index: number
│     ARIA (set by parent):
│       role="tab", aria-selected, aria-controls, tabindex, aria-disabled
│     Click handler delegates to parent's _onTabClick
│
└── CoreTabpanel (Light DOM — appears in consumer's CSS)
      Attrs (reflected): id (consumer-set), hidden (set by parent)
      Properties (read-only, set by parent):
        selected: boolean
        index: number
        tabId: string
      ARIA (set by parent):
        role="tabpanel", aria-labelledby, tabindex="0"
```

### Why this split

Matches the W3C ARIA APG tablist pattern exactly: `tablist` is the container (Shadow DOM here to encapsulate the indicator chrome), `tab` and `tabpanel` are siblings of each other inside the tablist's scope. Light DOM for `<core-tab>` + `<core-tabpanel>` so consumers can apply their own CSS to tab text and panel content without piercing Shadow DOM.

### File structure

```
src/tabs/
├── tabs.ts                # <core-tabs> Shadow DOM container
├── tabs.test.ts
├── tab.ts                 # <core-tab> light DOM trigger
├── tab.test.ts
├── tabpanel.ts            # <core-tabpanel> light DOM panel
├── tabpanel.test.ts
├── index.ts               # re-exports + side-effect imports
└── tabs_example_test.ts   # AX rule (4 examples)
```

## 4. Component APIs

### 4.1 `<core-tabs>`

```html
<core-tabs selected-index="0" orientation="horizontal" activation="auto">
  <core-tab for="general">General</core-tab>
  <core-tab for="account">Account</core-tab>
  <core-tab for="billing" disabled>Billing</core-tab>
  <core-tab for="security">Security</core-tab>

  <core-tabpanel id="general">General settings here.</core-tabpanel>
  <core-tabpanel id="account">Account settings.</core-tabpanel>
  <core-tabpanel id="billing">Billing (disabled in nav).</core-tabpanel>
  <core-tabpanel id="security">Security settings.</core-tabpanel>
</core-tabs>
```

**Attributes (reflected):**
- `selected-index` (number, default `0`) — bidirectional; reflects on selection change
- `orientation="horizontal|vertical"` (default `horizontal`)
- `activation="auto|manual"` (default `auto`)

**Properties (not reflected):**
- `selectedTab: CoreTab | null` (read-only getter)
- `selectedPanel: CoreTabpanel | null` (read-only getter)

**Methods:**
- `select(indexOrTab: number | CoreTab): void` — programmatic activation
- `refresh(): void` — force re-read of slotted children (after dynamic insertion)

**Events:**
- `core-tab-change` — cancellable; detail `{ index, previousIndex, tab, panel }`

**Slots:** default — accepts mixed `<core-tab>` + `<core-tabpanel>` children in any order.

**Parts:**
- `tablist` — flex container for the tab strip
- `indicator` — sliding active indicator
- `panels` — wrapper around the panels region

### 4.2 `<core-tab>`

```html
<core-tab for="account" disabled>Account</core-tab>
```

Light DOM. Click handler delegates to parent `<core-tabs>`.

**Attributes (reflected):**
- `for` (optional string) — explicit pairing with a `<core-tabpanel id="...">`
- `disabled` (boolean) — non-interactive

**Properties (read-only):**
- `selected: boolean` — set by parent when this tab is active
- `index: number` — slot position (raw, including disabled siblings)

**ARIA wiring (set by parent):**
- `role="tab"`
- `aria-selected="true|false"` mirrors `selected` property
- `aria-controls="<panel-id>"` (consumer-set OR auto-generated)
- `aria-disabled="true"` when `disabled`
- `tabindex="0"` when selected, `tabindex="-1"` otherwise (roving)

**Render:** light DOM — children pass through. The element itself owns the click listener.

### 4.3 `<core-tabpanel>`

```html
<core-tabpanel id="account">
  <h2>Account settings</h2>
  <p>Manage your account here.</p>
</core-tabpanel>
```

Light DOM. Visibility toggled by parent via native `hidden` attribute.

**Attributes (reflected):**
- `id` (optional, consumer-set) — paired with `<core-tab for="...">`. If absent, parent auto-generates a stable ID.
- `hidden` (boolean, set by parent) — true when this panel is not active

**Properties (read-only):**
- `selected: boolean`
- `index: number`
- `tabId: string` — consumer's `id` or auto-generated

**ARIA wiring (set by parent):**
- `role="tabpanel"`
- `aria-labelledby="<tab-id>"`
- `tabindex="0"` — entire panel is in tab order so keyboard users can Tab from active tab into panel content

### 4.4 Pairing resolution

Algorithm (run on every `slotchange`):

1. Collect all `<core-tab>` children → array `_tabs`
2. Collect all `<core-tabpanel>` children → array `_panels`
3. For each tab with a non-empty `for` attribute, look up matching `<core-tabpanel id="X">`. If found, mark both as "explicitly paired" and add `{tab, panel}` to `_pairs`.
4. For remaining unpaired tabs (in slot order), pair with remaining unpaired panels (in slot order) by residual index.
5. Any leftover tabs without panels get `aria-controls=""` (degenerate case — render but inert). Any leftover panels without tabs are not displayed but exist in DOM.
6. Generate stable IDs for panels without consumer-set `id`: `core-tabpanel-{tabsUid}-{panelIndex}`.
7. Generate stable IDs for tabs: `core-tab-{tabsUid}-{tabIndex}` (used for `aria-labelledby` on the panel side).

### 4.5 Auto-id generation

When consumer omits IDs, the parent generates per-instance stable IDs (format `core-tabpanel-{tabsUid}-{index}` and `core-tab-{tabsUid}-{index}`). The `tabsUid` is incremented from a module-level counter, ensuring uniqueness across multiple `<core-tabs>` instances on the same page.

Consumer-set IDs always take precedence — auto-generation only fills in gaps.

## 5. Behaviour details

### 5.1 Slotchange handling

`<core-tabs>` listens for `slotchange` on its default slot. On every change:

1. Re-read all slotted `<core-tab>` and `<core-tabpanel>` children
2. Re-run pairing resolution (§4.4)
3. Auto-generate IDs for unpaired panels (§4.5)
4. Apply ARIA (`role`, `aria-selected`, `aria-controls`, `aria-labelledby`, `tabindex`)
5. Apply `hidden` attribute to inactive panels (remove from active)
6. Apply roving `tabindex` (`0` on selected, `-1` on others)
7. Reposition indicator via `_repositionIndicator()` (CSS transform/width based on selected tab's `getBoundingClientRect`)
8. Clamp `selectedIndex` to `[0, tabCount - 1]` if children shrank — clamping fires `core-tab-change` if the index changed

### 5.2 Indicator positioning

```css
[part="indicator"] {
  position: absolute;
  bottom: 0;
  height: var(--core-tabs-indicator-thickness);
  background: var(--core-tabs-indicator-color);
  transition: transform 200ms cubic-bezier(0.4, 0, 0.2, 1),
              width 200ms cubic-bezier(0.4, 0, 0.2, 1);
}
:host([orientation="vertical"]) [part="indicator"] {
  bottom: auto;
  top: 0;
  left: auto;
  right: 0;
  width: var(--core-tabs-indicator-thickness);
  /* height + transform-Y set by JS */
}
```

JS `_repositionIndicator()`:
- Measures active tab's `getBoundingClientRect()` relative to tablist
- For horizontal: sets `transform: translateX(Npx)` + `width: Mpx`
- For vertical: sets `transform: translateY(Npx)` + `height: Mpx`

**happy-dom:** `getBoundingClientRect()` returns zeros — indicator JS positioning becomes a no-op (writes `transform: translateX(0px); width: 0px;`). Tests assert the indicator part exists but not pixel position. Real-browser visual fidelity covered by Playwright sweep.

### 5.3 Keyboard navigation

Roving tabindex pattern. Only the selected tab is in the tab order (`tabindex="0"`); others are `tabindex="-1"`. Arrow keys move focus within the tablist without leaving the tab order entirely.

| Key | Horizontal | Vertical | Effect |
|---|---|---|---|
| `ArrowRight` | move next | — | focus + (auto: activate) next non-disabled tab; wrap at boundaries |
| `ArrowLeft` | move prev | — | focus + (auto: activate) prev non-disabled tab; wrap at boundaries |
| `ArrowDown` | — | move next | as ArrowRight in horizontal |
| `ArrowUp` | — | move prev | as ArrowLeft in horizontal |
| `Home` | first | first | focus + (auto) activate first non-disabled tab |
| `End` | last | last | focus + (auto) activate last non-disabled tab |
| `Space` | activate | activate | manual mode only — activate currently focused tab |
| `Enter` | activate | activate | manual mode only — activate currently focused tab |

In `activation="auto"` mode, focus and activation are the same event. In `activation="manual"` mode, arrow keys move focus only; Space/Enter activates the focused tab.

### 5.4 Disabled handling

`<core-tab disabled>`:
- Click handler returns early (no-op)
- Keyboard nav (`_nextEnabledIndex(from, direction)`) skips disabled tabs
- `aria-disabled="true"` set automatically
- If all tabs are disabled (degenerate), keyboard nav is a no-op (no infinite loop)

If the currently-selected tab becomes disabled mid-session (e.g. consumer toggles `disabled` attr), selection persists — `disabled` only affects future nav, not current selection. Consumer can call `tabs.select(...)` to move selection to an enabled tab.

### 5.5 Activation modes

**`activation="auto"` (default — W3C automatic):**
- Arrow keys → focus + activate simultaneously
- Panel switches as user arrows through
- Best for tabs with light content
- W3C-recommended default per ARIA APG

**`activation="manual"`:**
- Arrow keys → focus moves only
- Space or Enter on the focused tab → activate
- Best for tabs with heavy content (avoids triggering N panel renders during keyboard scan)
- W3C-valid alternative per ARIA APG

Both modes fire `core-tab-change` on activation. Both honour `preventDefault()`.

### 5.6 Programmatic API

```ts
const tabs = document.querySelector('core-tabs');

tabs.selectedIndex;             // current index (number, via attr)
tabs.selectedTab;               // CoreTab | null
tabs.selectedPanel;             // CoreTabpanel | null

tabs.select(2);                 // activate by index
tabs.select(elTab);             // activate by element ref
tabs.refresh();                 // re-read children after dynamic insertion

tabs.addEventListener('core-tab-change', (e) => {
  if (e.detail.index === 2 && !userAllowedAccess) {
    e.preventDefault(); // block activation
  }
});
```

### 5.7 Cancellable activation

`core-tab-change` is cancellable. Consumer `preventDefault()` blocks the activation:
- `selectedIndex` does not change
- Active tab's `aria-selected` does not change
- Panel visibility does not change
- Indicator does not move

Use case: "confirm before leaving the current tab if there's unsaved work."

### 5.8 Accessibility

| Element | Role | Required ARIA |
|---|---|---|
| `[part="tablist"]` (Shadow root child) | `tablist` | `aria-orientation="horizontal\|vertical"` |
| `<core-tab>` | `tab` | `aria-selected`, `aria-controls`, `aria-disabled`, `tabindex` (roving) |
| `<core-tabpanel>` | `tabpanel` | `aria-labelledby`, `tabindex="0"`, `hidden` when inactive |

All ARIA attributes set automatically by the parent based on state. Consumer doesn't need to wire anything.

`aria-orientation` on the tablist reflects the `orientation` attribute. Screen readers announce keyboard nav direction correctly.

### 5.9 happy-dom guardrails

- `slotchange` fires correctly in happy-dom ✓ — slot-based reading works
- `MutationObserver` fires on attribute changes ✓ — backup signal if slotchange misses
- `getBoundingClientRect()` returns zeros — indicator JS positioning is a no-op; tests assert attribute state + part existence rather than pixel coordinates
- Native `<input>` interactions not used (tabs are not form-associated)
- Custom element registration order managed by `index.ts` imports — `tab.ts` + `tabpanel.ts` imported before `tabs.ts` so children are defined when parent connects
- Keyboard event dispatch works ✓ — keyboard nav tests can simulate ArrowLeft/Right/Up/Down/Home/End/Space/Enter

Every behaviour relying on real-browser semantics (indicator slide animation, focus ring visibility, actual focus moves with keyboard) is gated with `if (...) return;` early-returns rather than `it.skip()`. Each early-return commented `// Playwright sweep covers …`.
