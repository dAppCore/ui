# CoreUI v0.7 — Form Components (Design)

> *Status:* Approved (brainstorm 2026-05-16). *Authors:* Snider, Cladius. *Track:* v0.7 of the RFC roadmap (RFC §16).
> *Foundation:* Builds on v0.2 utils + v0.5 primitives + v0.6 router. Spec at `docs/superpowers/specs/2026-05-16-coreui-v0.5-primitives-design.md`.

## 1. Mission

Ship six brand-neutral form-input Web Components — `<core-input>`, `<core-textarea>`, `<core-select>`, `<core-checkbox>`, `<core-radio>`, `<core-radio-group>` — with the full Constraint Validation API surface and ElementInternals form participation. These complete the agent-app form story: any agent building a sign-up form, settings panel, or data-entry surface gets validated inputs that play with native `<form>` semantics without reinventing.

This is the **v0.7 slice** of the components track. v0.3 (`<core-data-table>`) and v0.8 (surface components) get their own brainstorms.

## 3. Locked design decisions

| Decision | Choice |
|---|---|
| `<core-select>` implementation | **Native inner `<select>`** (not custom listbox). Slotted `<option>` children re-projected into the Shadow-DOM-internal `<select>`. Rich-content options ship later via a separate `<core-combobox>` primitive after v0.8 popover lands. |
| Constraint Validation API depth | **Full surface.** Every primitive exposes `setValidity`, `validity`, `validationMessage`, `willValidate`, `checkValidity`, `reportValidity`, `setCustomValidity`. `formResetCallback` + `formDisabledCallback` + `formStateRestoreCallback` wired. |
| Radio structure | **Both `<core-radio>` and `<core-radio-group>`.** Styled radio matches `<core-toggle>` visual language; group manages single-selection via ElementInternals on itself, NOT on children. |
| Slot usage | **Mixed.** Icons via attribute (`leading-icon` looks up the icon registry from v0.5). Helper text + error messages via Shadow-DOM `<slot name="hint">` and `<slot name="error">`. |
| Render mode | **Shadow DOM.** RFC §4 reserves Shadow for "backdrop isolation"; slot distribution is a similar legit reason. CSS custom properties pierce shadow boundaries; tokens still flow. Skin layers use real `::part()` pseudo-element (not the attribute-selector workaround from v0.5). |
| Execution | **Greenfield rewrite.** Source `lethean-input.ts` / `lethean-select.ts` / `lethean-radio-group.ts` are behaviour references, not starting points. |
| Pacing | **One bundled PR** with 12 logical-cluster commits. |

```
├── core-form.test.ts           Existing
```

**No `.css` siblings.** Shadow DOM means default styles ship inside the component via `static styles = css\`...\``. Skin layers reach internals via real `::part()` pseudo-element.

**Counts:** 6 new primitive `.ts` + 6 test files + `_shared/` (3) + AX example test (1) = **16 new files**. Plus 2 modified existing files (`core-form.ts`, `core-field.ts`) in the cleanup commit.

**`package.json` `exports`** gains: `./forms/input`, `./forms/textarea`, `./forms/select`, `./forms/checkbox`, `./forms/radio`, `./forms/radio-group`, `./forms/_shared/form-element` (7 entries). No CSS exports needed.

**Why `src/forms/` rather than extending `src/primitives/`** — forms are an architectural tier apart from primitives:

- Primitives are light DOM; forms are Shadow DOM.
- Primitives use ElementInternals minimally (button + toggle only); forms use the full Constraint Validation surface.
- The pre-existing `<core-form>` and `<core-field>` live here already.

Keeping them in `src/forms/` makes the contract clear: `@dappcore/ui/primitives` = light DOM behavioural primitives, `@dappcore/ui/forms` = Shadow DOM form-associated inputs.

## 5. Shared infrastructure (`src/forms/_shared/`)

### 5.1 `CoreFormElement` base class

Single shared base class extending `LitElement` (NOT `CoreElement` — that one is light DOM and v0.7 forms are Shadow DOM). Provides ElementInternals attachment + the full Constraint Validation API surface.

```ts
// src/forms/_shared/form-element.ts
import { LitElement } from 'lit';

export abstract class CoreFormElement extends LitElement {
  static formAssociated = true;

  protected _internals: ElementInternals | null;

  abstract name: string;
  abstract disabled: boolean;
  abstract required: boolean;

  constructor() {
    super();
    this._internals = typeof this.attachInternals === 'function'
      ? this.attachInternals()
      : null;
  }

  // ── Constraint Validation API surface ──

  get form(): HTMLFormElement | null { return this._internals?.form ?? null; }
  get validity(): ValidityState | null { return this._internals?.validity ?? null; }
  get validationMessage(): string { return this._internals?.validationMessage ?? ''; }
  get willValidate(): boolean { return this._internals?.willValidate ?? false; }

  checkValidity(): boolean { return this._internals?.checkValidity() ?? true; }
  reportValidity(): boolean { return this._internals?.reportValidity() ?? true; }

  setCustomValidity(message: string): void {
    this._internals?.setValidity(message ? { customError: true } : {}, message);
  }

  protected _setValidity(
    flags: ValidityStateFlags,
    message?: string,
    anchor?: HTMLElement,
  ): void {
    this._internals?.setValidity(flags, message, anchor);
  }

  protected _setFormValue(value: FormDataEntryValue | null): void {
    this._internals?.setFormValue(value);
  }

  // ── Form lifecycle callbacks (subclass overrides where applicable) ──

  formResetCallback(): void { /* default no-op; subclass overrides */ }
  formDisabledCallback(disabled: boolean): void {
    (this as any).disabled = disabled;
  }
  formStateRestoreCallback(
    _state: FormDataEntryValue | null,
    _mode: 'restore' | 'autocomplete',
  ): void { /* subclass overrides for form state restoration */ }
}
```

happy-dom guard built into the constructor (same pattern as v0.5 button + toggle). All public Validation API methods return safe defaults when `_internals` is null.

### 5.2 Test discipline for the base

```ts
class TestInput extends CoreFormElement {
  name = ''; disabled = false; required = false;
}
customElements.define('test-form-input', TestInput);

describe('CoreFormElement', () => {
  it('attaches internals when available (or stores null under happy-dom)', () => { /* asserts el._internals is ElementInternals | null */ });
  it('exposes the full Constraint Validation API surface (or safe defaults when internals is null)', () => { /* asserts checkValidity, reportValidity, validity, etc are all callable */ });
  it('formDisabledCallback mirrors the form-level disable to subclass `disabled`', () => { /* call el.formDisabledCallback(true); assert el.disabled === true */ });
});
```

### 5.3 Why this base class works under Shadow DOM

`CoreFormElement` does NOT override `createRenderRoot()`. It uses Lit's default Shadow DOM attachment. This means:

- `static styles = css\`...\`` works natively (no `<style>` in the body).
- `<slot>` distribution works as intended.
- `::part()` skinning works as a real Shadow DOM pseudo-element.
- `--core-*` tokens still cascade (custom properties pierce shadow).

### 5.4 What's NOT in `_shared/`

- No icon registry — v0.7 forms use the `<core-icon>` primitive from v0.5 (consumer drops it inside via `leading-icon` attribute resolution).
- No focus-trap or popover logic — those land in v0.8 surfaces.
- No theming helpers — CSS custom properties pierce shadow naturally; the primitives' `static styles` consume `--core-*` tokens directly.

## 6. Per-primitive surface

All six extend `CoreFormElement`. Shadow DOM, `static styles` inside, real `::part()` skinning.

### 6.1 `<core-input>`

| Property | Detail |
|---|---|
| Attrs | `type` (`text \| email \| password \| number \| tel \| url \| search`; default `text`), `name`, `value` (reflected), `placeholder`, `required`, `disabled`, `readonly`, `pattern`, `minlength`, `maxlength`, `min`/`max`/`step` (number), `autocomplete`, `leading-icon` (icon name lookup), `trailing-icon`, `size` (`sm \| md \| lg`; default `md`) |
| Slots | `slot="hint"`, `slot="error"` |
| Parts | `base`, `input`, `leading`, `trailing`, `hint`, `error` |
| Events | `core-input`, `core-change` (composed, bubbles) wrapping inner native events with `detail: { value }` |
| Validation | Mirrors inner `<input>`'s `validity` to ElementInternals on every change. Native constraints (required, pattern, minlength, maxlength, min, max, step) work natively. |
| Vars | `--core-input-{height, padding-x, font-size, bg, fg, border-color, border-color-focus, border-color-invalid, radius, hint-color, error-color}` |

### 6.2 `<core-textarea>`

| Property | Detail |
|---|---|
| Attrs | `name`, `value`, `placeholder`, `required`, `disabled`, `readonly`, `rows` (default 3), `cols`, `minlength`, `maxlength`, `wrap` (`soft \| hard \| off`), `resize` (`none \| vertical \| horizontal \| both`; default `vertical`), `autocomplete`, `size` (`sm \| md \| lg`) |
| Slots | `slot="hint"`, `slot="error"` |
| Parts | `base`, `textarea`, `hint`, `error` |
| Events | `core-input`, `core-change` |
| Validation | required + minlength + maxlength mirrored from inner `<textarea>` |
| Vars | `--core-textarea-{padding, font-size, bg, fg, border-color, border-color-focus, border-color-invalid, radius, min-height, hint-color, error-color}` |

### 6.3 `<core-select>`

| Property | Detail |
|---|---|
| Attrs | `name`, `value` (reflected), `required`, `disabled`, `size` (`sm \| md \| lg`), `multiple`, `leading-icon` |
| Slots | default (consumer's `<option>` children — re-projected into the Shadow-DOM-internal `<select>`), `slot="hint"`, `slot="error"` |
| Parts | `base`, `select`, `leading`, `hint`, `error` |
| Events | `core-change` with `detail: { value, selectedValues }` (`selectedValues` populated only when `multiple` is set) |
| Validation | required (ValueMissing if value is empty after user interaction) |
| Vars | `--core-select-{height, padding-x, font-size, bg, fg, border-color, border-color-focus, border-color-invalid, radius, hint-color, error-color}` |

**Option re-projection mechanics:** consumer writes `<core-select><option value="...">Label</option>...</core-select>`. The `<option>` elements are direct children of the host (not in a slot, since `<select>` only accepts `<option>` and `<optgroup>` children directly). On `connectedCallback` and via a `MutationObserver`, we move them into the Shadow-DOM-internal `<select>` element. Inner `<select>` and the live host options stay in sync via the observer.

**Multiple-select FormData serialization:** when `multiple` is set and multiple options are selected, ElementInternals receives the value via a `FormData` instance built with one append per selected value sharing the same `name`. This matches native `<select multiple name="x">` behaviour: `FormData.getAll('x')` returns the array; `FormData.get('x')` returns the first.

### 6.4 `<core-checkbox>`

| Property | Detail |
|---|---|
| Attrs | `name`, `value` (default `"on"`), `checked` (reflected), `indeterminate`, `required`, `disabled`, `size` (`sm \| md \| lg`) |
| Slots | default (label text), `slot="hint"`, `slot="error"` |
| Parts | `base`, `box`, `tick`, `label`, `hint`, `error` |
| Events | `core-change` with `detail: { checked }` |
| Validation | required → ValueMissing if not checked |
| Vars | `--core-checkbox-{size, bg, border-color, border-color-focus, tick-color, bg-checked, border-color-checked, radius}` |

Different from `<core-toggle>` (v0.5): toggle is a switch (track + thumb sliding), checkbox is a box with a tick. Semantically and visually distinct.

**Indeterminate state:** `indeterminate` is a tri-state visual hint (typically used for "some-but-not-all" parent checkboxes). It does NOT contribute to `FormData` — only `checked` does. When the user clicks an indeterminate checkbox, it transitions to checked + indeterminate=false (native browser behaviour).

### 6.5 `<core-radio>`

| Property | Detail |
|---|---|
| Attrs | `name`, `value`, `checked` (reflected), `required`, `disabled`, `size` (`sm \| md \| lg`) |
| Slots | default (label text) |
| Parts | `base`, `dot`, `label` |
| Events | `core-change` with `detail: { checked, value }` |
| Validation | Delegated to `<core-radio-group>` (see §6.6). Standalone `<core-radio>` outside a group doesn't validate. |
| Vars | `--core-radio-{size, bg, border-color, dot-color, dot-color-checked, bg-checked, border-color-checked}` |

### 6.6 `<core-radio-group>`

| Property | Detail |
|---|---|
| Attrs | `name`, `value` (reflected), `required`, `disabled`, `orientation` (`vertical \| horizontal`; default `vertical`) |
| Slots | default (consumer's `<core-radio>` children), `slot="hint"`, `slot="error"` |
| Parts | `base`, `group` (the radio list wrapper), `hint`, `error` |
| Events | `core-change` with `detail: { value }` |
| Validation | required → ValueMissing if no radio selected. ElementInternals wired on the group, NOT individual radios. |
| Vars | `--core-radio-group-{gap, hint-color, error-color}` |

**Group mechanics:** queries `<core-radio>` children via `slotchange` event + `MutationObserver` for late-added children. Propagates `name` and `disabled` to each radio (group's `name` overwrites any radio's `name` to enforce single-group semantics). Listens for `core-change` on children; updates `value`; emits its own `core-change`; un-checks siblings. Single source of selected value lives on the group.

## 7. Validation display pattern (applies to all 6)

- The inner native input / textarea / select carries native validity. The CSS `:user-invalid` pseudo-class on the inner element fires automatically **after** the user has interacted with it (W3C standard behaviour). The `--core-<name>-border-color-invalid` var is keyed to this state via `[part="input"]:user-invalid { border-color: var(--core-input-border-color-invalid); }`.
- The `error` slot is shown **unconditionally** when populated — consumer decides when to fill it (post-submit, server-side error, etc.). Consumer can use `:user-invalid` styling in their CSS to hide-until-touched if they want that flow.
- `setCustomValidity('Server says no')` propagates to ElementInternals and to the inner native input, enabling browser-default `reportValidity()` tooltips.

## 11. Risk register

| Risk | Mitigation |
|---|---|
| Shadow DOM + tokens — does theming break? | CSS custom properties pierce shadow boundaries natively. Tokens are 100% custom-property-based since v0.2. Verified by `<core-button>` ElementInternals tests already working. |
| `::part()` pseudo-element browser support | Supported in all evergreens since 2020. Same baseline as ElementInternals. |
| happy-dom ElementInternals gaps | Same workaround as v0.5: `typeof attachInternals` guard + integration tests skipped where they need real EI. |
| `<core-select>` option re-projection race | `MutationObserver` + `slotchange` cover both initial parse and dynamic add/remove. Tests in `select.test.ts` lock this. |
| `<core-radio-group>` name propagation conflicts | Group's `name` overwrites any radio's `name` to enforce single-group semantics. Documented behaviour. |
| Inner native input losing focus on consumer-driven re-renders | Lit's keyed-children handling preserves the inner element across renders. Test this explicitly in `input.test.ts`. |
| `:user-invalid` browser compatibility | Supported in Firefox / Safari since 2022, Chrome since 2024. Documented minimum browser baseline; older browsers fall back to never-styled-invalid (no error UI but no broken UI either). |
| Pre-existing crypto/forms typecheck errors | Fixed in commit 1 of this PR. If the fix turns out non-trivial, escalate before continuing. |
