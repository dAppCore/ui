# CoreUI v0.8 — Surfaces (Dialog / Drawer / Popover / Tooltip) Design

**Status:** Approved (brainstorm 2026-05-17)
**Spec file:** `docs/superpowers/specs/2026-05-17-coreui-v0.8-surfaces-design.md`
**Plan file:** `docs/superpowers/plans/2026-05-17-coreui-v0.8-surfaces.md` (to be written)
**Target version:** `@dappcore/ui@0.8.0`
**Prior milestone:** v0.7.0 (forms layer, commit `1020ac8`)

---

## 1. Goal

Ship the **surfaces** tier of CoreUI — the four overlay primitives every app needs: modal dialog, side drawer, anchored popover, hover/focus tooltip. Brand-neutral, Shadow DOM, platform-API-first, zero deps other than Lit. Consumer-overridable via CSS variables and `::part()`.

This unblocks v0.3 (`<core-data-table>` row actions) and the eventual `<core-combobox>` / `<core-listbox>` (both need anchored surfaces).

---

## 2. Scope

### In scope

- `<core-dialog>` — modal and non-modal modal element with size variants
- `<core-drawer>` — edge-attached panel with `side="start|end|top|bottom"`
- `<core-popover>` — anchored floating panel with 12-position placement
- `<core-tooltip>` — hover/focus-triggered descriptor, a11y-compliant
- Two shared base classes: `CoreOverlayElement`, `CoreAnchoredElement`
- Two pure utilities: `focus-trap.ts`, `anchor-position.ts`
- Animation tokens, dismissal patterns, focus management, a11y wiring
- AX-rule example tests, integration tests, README + CLAUDE.md docs

### Out of scope (deferred)

- **Arrow indicators** on tooltip/popover — positioning math is non-trivial; v0.8.1 candidate
- **View Transitions API** integration — Chromium-only in 2026; revisit when Safari/Firefox ship
- **Touch long-press tooltip** — fails WCAG ergonomics; relying on hover+focus baseline
- **`<core-combobox>` / `<core-listbox>`** — depend on popover landing, separate milestone
- **`<core-data-table>`** — RFC §8 seed component, its own milestone (v0.3)
- **Toast / snackbar / notification** — different paradigm (auto-dismiss, queueing), v0.9 candidate

## 3. Architecture

### Layer position

v0.8 surfaces sit on the v0.2 foundation (`tokens`, `colour`, `a11y`, `dom`, `animation`, `platform`). Independent of v0.5 primitives and v0.7 forms — consumers can pull v0.8 without either.

### Base class hierarchy

```
LitElement
├── CoreOverlayElement (abstract, Shadow DOM)
│   ├── attachInternals + role/aria-modal wiring (happy-dom-guarded)
│   ├── open / show() / showModal() / close(returnValue?) / toggle()
│   ├── modal / open / closedby / restoreFocus state mgmt
│   ├── data-state="closed|opening|open|closing"
│   ├── ESC + backdrop-click handlers
│   ├── focus trap (via focus-trap.ts) for non-modal mode
│   ├── native focus trap for .showModal() modal mode
│   ├── return-focus on close
│   ├── used by: <core-dialog>, <core-drawer>
│
└── CoreAnchoredElement (abstract, Shadow DOM)
    ├── attachInternals + role wiring (happy-dom-guarded)
    ├── anchor attribute (CSS selector) resolution
    ├── anchorElement property (programmatic ref)
    ├── placement → position-area CSS generation
    ├── JS positioning fallback (anchor-position.ts) when CSS unsupported
    ├── light-dismiss via Popover API
    ├── data-state="closed|opening|open|closing"
    ├── used by: <core-popover>, <core-tooltip>
```

### Platform APIs used

- **Popover API** (`popover` attribute + `showPopover()/hidePopover()`) — baseline since 2024 in all evergreen browsers
- **`<dialog>` element** — modal/non-modal + native focus trap + `::backdrop`, baseline since 2022
- **CSS Anchor Positioning** (`anchor-name`, `position-area`) — Chromium-only in 2026, JS fallback for Safari/Firefox
- **`prefers-reduced-motion`** — animations bypassed under the media query

│   ├── anchored-element.ts       # CoreAnchoredElement base

│   └── anchor-position.ts        # CSS.supports check + JS fallback positioner
├── dialog.ts                     # <core-dialog>
├── drawer.ts                     # <core-drawer>

## 4. Component APIs

### 4.1 `<core-dialog>`

```html
<core-dialog modal open size="md" closedby="closerequest">
  <h2 slot="header">Confirm deletion</h2>
  <p>This action can't be undone.</p>
  <div slot="footer">
    <core-button onclick="this.closest('core-dialog').close('cancel')">Cancel</core-button>
    <core-button variant="danger" onclick="this.closest('core-dialog').close('confirm')">Delete</core-button>
  </div>
</core-dialog>
```

**Attributes**
- `open` (bool) — drives state
- `modal` (bool, default `true`) — `.showModal()` vs `.show()` path
- `size="sm|md|lg|xl|full"` (default `md`) — maps to widths via tokens (`--core-dialog-width-sm` ... `--core-dialog-width-full`)
- `closedby="any|none|closerequest"` (default `closerequest`)

**Methods** — mirror native `<dialog>`
- `show(): void`
- `showModal(): void`
- `close(returnValue?: string): void`
- `toggle(): void` — `show()` if closed, `close()` if open

**Properties**
- `returnValue: string` — written by `close(value)`, readable after close

**Events**
- `core-dialog-open` — fires after `data-state` reaches `open`
- `core-dialog-close` (detail: `{ returnValue: string }`) — fires after `data-state` reaches `closed`
- `core-dialog-cancel` — cancellable, fires when ESC requested close; consumer `preventDefault()` blocks dismissal

**Slots**: default (body), `header`, `footer`
**Parts**: `dialog`, `header`, `body`, `footer`, `backdrop`

### 4.2 `<core-drawer>`

```html
<core-drawer modal open side="end">
  <h2 slot="header">Cart (3 items)</h2>
  <core-list>…</core-list>
  <div slot="footer"><core-button>Checkout</core-button></div>
</core-drawer>
```

**Attributes**
- `open`, `modal` (default `true`)
- `side="start|end|top|bottom"` (default `end`) — logical sides, respects RTL
- `closedby` (default `any` — backdrop click conventional for drawers)

**Methods / Properties / Events**: same shape as `<core-dialog>` with `core-drawer-*` event prefix
**Slots**: default, `header`, `footer`
**Parts**: `drawer`, `header`, `body`, `footer`, `backdrop`

### 4.3 `<core-popover>`

```html
<core-button id="more-btn">More</core-button>
<core-popover anchor="#more-btn" placement="bottom-start" offset="8">
  <core-menu>
    <core-menu-item>Edit</core-menu-item>
    <core-menu-item>Delete</core-menu-item>
  </core-menu>
</core-popover>
```

**Attributes**
- `open`
- `anchor` (CSS selector string) — queried via `getRootNode().querySelector(selector)`, falling back to `document.querySelector(selector)` if not found in the local root. Resolved on connect + on attribute change.
- `placement` (default `bottom`) — 12 values:
  - `top` · `top-start` · `top-end`
  - `bottom` · `bottom-start` · `bottom-end`
  - `start` · `start-start` · `start-end`
  - `end` · `end-start` · `end-end`
- `offset` (px integer, default `4`)
- `closedby` (default `any`)
- `autofocus` (bool, default `false`) — if set, first focusable inside popover focuses on open

**Properties**
- `anchorElement: HTMLElement | null` — programmatic anchor (overrides `anchor` attribute)

**Methods**: `show()`, `hide()`, `toggle()`
**Events**: `core-popover-open`, `core-popover-close`
**Slots**: default
**Parts**: `popover`

### 4.4 `<core-tooltip>`

```html
<core-button id="save-btn"><core-icon name="save"/></core-button>
<core-tooltip anchor="#save-btn" placement="top" delay-in="700">Save (⌘S)</core-tooltip>
```

**Attributes**
- `open` (programmatic — hover/focus drive open/close internally)
- `anchor` (CSS selector — same resolution semantics as `<core-popover>`: local root first, then document)
- `placement` (default `top`)
- `offset` (default `6`)
- `delay-in` (ms, default `700`)
- `delay-out` (ms, default `0`)

**Properties**: `anchorElement: HTMLElement | null`
**Methods**: `show()`, `hide()` (programmatic open/close, bypasses delay)
**Events**: `core-tooltip-open`, `core-tooltip-close`
**Slots**: default
**Parts**: `tooltip`

**Auto-binding behaviour:**
- On `connectedCallback`: resolve `anchorElement`, attach `mouseenter`/`focusin`/`mouseleave`/`focusout` listeners
- On `disconnectedCallback`: detach listeners
- On `anchor` attribute change or `anchorElement` setter: detach from old, attach to new
- `aria-describedby` on anchor automatically set/cleared (preserves any prior value, restores on disconnect)

**Never autofocuses** — tooltips describe other content; stealing focus is hostile.

## 5. Shared concerns

### 5.1 State machine

```
closed → opening → open → closing → closed
```

| State | `open` attr | `data-state` | Rendered? | Interactive? |
|---|---|---|---|---|
| `closed` | absent | `closed` | `display: none` | no |
| `opening` | present | `opening` | yes, transitioning in | yes |
| `open` | present | `open` | yes, at rest | yes |
| `closing` | absent | `closing` | yes, transitioning out | no (pointer-events: none) |

**Transition driver:** `transitionend` / `animationend` on `:host` advances opening→open and closing→closed. Fallback timer (`--core-overlay-duration + 50ms`) handles cases where no transition is defined or `prefers-reduced-motion` zeroes the duration. happy-dom never fires transition events, so the fallback timer is the de facto path under tests.

### 5.2 Animations

Default chrome per component:

| Component | Transition |
|---|---|
| dialog | fade + scale(0.96 → 1) |
| drawer | slide from `side` edge |
| popover | fade + 2px offset from anchor |
| tooltip | fade only (no transform near anchor — avoids jitter) |

```css
:host {
  --core-overlay-duration: 200ms;
  --core-overlay-easing: cubic-bezier(0.4, 0, 0.2, 1);
  --core-overlay-backdrop: oklch(0% 0 0 / 0.5);
  --core-overlay-z-index: 100;
}
:host([data-state="opening"]),
:host([data-state="closing"]) {
  transition: opacity var(--core-overlay-duration) var(--core-overlay-easing),
              transform var(--core-overlay-duration) var(--core-overlay-easing);
}
:host([data-state="closed"]) { display: none; }
:host([data-state="closing"]) { pointer-events: none; }

@media (prefers-reduced-motion: reduce) {
  :host { transition: none !important; }
}
```

Consumers override by restyling `:host([data-state=opening])` etc., or by setting `--core-overlay-duration: 0` to disable animation entirely.

### 5.3 Focus management

| Mode | Trap | Auto-focus first focusable | Return focus on close |
|---|---|---|---|
| Dialog / drawer, `modal` | native (browser via `.showModal()`) | yes | yes (snapshot at open) |
| Dialog / drawer, non-modal | via `focus-trap.ts` (only if `trap` attr present) | yes unless `noautofocus` | yes |
| Popover | none | only if `autofocus` attr | yes |
| Tooltip | none | never | n/a (no programmatic focus stolen) |

`focus-trap.ts` API:

```ts
export interface FocusTrap {
  activate(): void;
  deactivate(): void;
}
export function createFocusTrap(root: HTMLElement): FocusTrap;
```

- Activates: stores `document.activeElement`, finds focusables inside `root`, focuses first
- While active: intercepts `keydown` (Tab/Shift+Tab) and cycles focus inside `root`
- Deactivates: removes listener, returns focus to stored element if still in DOM
- Ignores: `[hidden]`, `[disabled]`, `[tabindex="-1"]`, `<input type="hidden">`

### 5.4 Dismissal patterns (closedby polyfill)

| Value | ESC | Backdrop click | Light-dismiss (Popover) | Close button (`[data-core-close]`) |
|---|---|---|---|---|
| `any` | ✓ | ✓ | ✓ | ✓ |
| `closerequest` | ✓ | — | — | ✓ |
| `none` | — | — | — | — |

Defaults: dialog = `closerequest` · drawer/popover/tooltip = `any`.

**Backdrop click detection** (dialog/drawer, modal mode only — non-modal has no backdrop): event listener on the host, fires `close()` if `event.target === hostEl` (clicks inside the body have a different target).

**Light-dismiss for popover/tooltip:** when `closedby="any"`, set `popover="auto"` so Popover API auto-dismisses on outside click. When `closedby="none|closerequest"`, set `popover="manual"` to suppress.

**Close-button convention:** any descendant with `[data-core-close]` attribute will close the surface when clicked. The host installs a single delegated click listener at connect time.

### 5.5 Accessibility

| Component | Role | aria-modal | aria-labelledby | Other |
|---|---|---|---|---|
| dialog | `dialog` | mirrors `modal` attr | auto-set if `<h1-h6>` in `header` slot; consumer overrides via attr | — |
| drawer | `dialog` | mirrors `modal` attr | same as dialog | — |
| popover | none (consumer sets `role` per content — typically `menu`, `dialog`, or `listbox`) | — | consumer sets if `role="dialog"` chosen | — |
| tooltip | `tooltip` | — | — | auto-wires `aria-describedby` on anchor |

**Tooltip aria-describedby lifecycle:**
1. On connect with anchor resolved: read anchor's current `aria-describedby`, store as previous value
2. Generate unique id for tooltip if not set
3. Append tooltip's id to anchor's `aria-describedby` (space-separated)
4. On disconnect or anchor swap: restore previous value (remove tooltip id from list)

**Reduced motion:** transition-duration zeroed via `@media (prefers-reduced-motion: reduce)`. State machine still advances; fallback timer still fires.

## 6. Anchor positioning (popover/tooltip detail)

### 6.1 Platform path (Chromium 2026)

The host element opts in to CSS Anchor Positioning. Anchor element gets `anchor-name: --core-anchor-{uid}` via inline style (one-time per anchor; uid generated per-component-instance). Host element gets `position-anchor: --core-anchor-{uid}` plus per-placement positioning rules.

For each of the 12 placements, the implementer chooses between `position-area` (sugar for cardinal placements) and direct `anchor()` function calls (for `-start`/`-end` variants needing precise edge alignment). The 12 placements behave as in floating-ui convention:

- `top` / `bottom` / `start` / `end` — popover on that side, centred on the orthogonal axis
- `top-start` / `top-end` — above, aligned to anchor's start or end edge (start = left in LTR)
- `bottom-start` / `bottom-end` — below, aligned to anchor's start or end edge
- `start-start` / `start-end` — to the start side, aligned to anchor's top or bottom edge
- `end-start` / `end-end` — to the end side, aligned to anchor's top or bottom edge

`offset` attribute applied as `margin` in the direction away from anchor (e.g. `margin-bottom` for `top` placements).

The exact CSS construction is implementer's call in the plan. The JS fallback is the authoritative reference for coordinates.

### 6.2 JS fallback path

`anchor-position.ts`:

```ts
export interface PositionResult {
  top: number;   // px from viewport
  left: number;
  placement: Placement; // may differ from requested if flipped
}
export type Placement = 'top' | 'top-start' | 'top-end' | 'bottom' | /* ... */;

export function supportsAnchorPositioning(): boolean;
export function computePosition(
  anchor: Element,
  surface: HTMLElement,
  placement: Placement,
  offset: number,
): PositionResult;
```

- `supportsAnchorPositioning()`: tests via probe element — `el.style.anchorName = '--x'; getComputedStyle(el).anchorName !== ''`. `CSS.supports('anchor-name', '--x')` returns true in happy-dom (incorrectly), so the probe path is the trusted detection
- `computePosition()`: `getBoundingClientRect()` on both, places surface at requested placement, clamps to viewport (flipping to opposite side if overflow)

When fallback is active, `CoreAnchoredElement` recomputes on: open, `window` resize, `window` scroll (throttled with `requestAnimationFrame`), anchor element mutation. Listeners attached with `AbortSignal` for clean teardown.

When CSS path is active, no JS positioning runs — browser handles repositioning automatically.

### 7.2 Integration tests

`tests/integration/surfaces/v08/`:

- **`modal-focus-trap.test.ts`** — focus stays inside modal dialog/drawer across tab cycles. Early-returns under happy-dom (`if (typeof HTMLDialogElement.prototype.showModal !== 'function' || !document.activeElement) return;`). Playwright-bound.
- **`anchored-positioning.test.ts`** — 12 placements → expected coords within ±2px; viewport clamping. Mostly happy-dom-friendly (geometry math is pure JS once anchor + surface rects are known).
- **`light-dismiss.test.ts`** — click outside dismisses popover/tooltip when `closedby="any"`; backdrop click dismisses dialog/drawer when `closedby="any"`. Early-returns where Popover API light-dismiss is needed.
- **`tooltip-a11y.test.ts`** — `aria-describedby` set on anchor when tooltip opens, cleaned up on close + on tooltip disconnect; hover and focus trigger pairs.

### 7.3 happy-dom guardrails inventory

- `<dialog>.show()/.showModal()/.close()` exist; no focus trap, no `::backdrop` rendering
- `popover` attribute + `showPopover()/hidePopover()` exist; no light-dismiss event
- `ElementInternals` absent (same as v0.7) — `attachInternals` guard required
- `CSS.supports('anchor-name', '--x')` returns true (incorrect) — probe via computed style instead
- `transitionend`/`animationend` never fire — fallback timer is the de facto path
- `getBoundingClientRect()` returns plausible values (zeroes if not laid out)

Every behaviour relying on real browser semantics is gated with `if (...) return;` early-returns rather than `it.skip()` — the test file structure documents what needs Playwright sweep.
